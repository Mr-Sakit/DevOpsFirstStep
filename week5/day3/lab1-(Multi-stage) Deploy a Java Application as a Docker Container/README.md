# Lab 1: (Multi-Stage) Deploy a Java Application as a Docker Container

## 📋 Overview

This lab demonstrates how to containerize a **Java Spring Boot e-commerce application** using a **multi-stage Docker build**. The multi-stage approach separates the build environment (Maven) from the runtime environment (JRE), resulting in a significantly smaller and more secure production image. The lab also covers troubleshooting a common build error and verifying the running application in a web browser.

---

## 🎯 Objectives

- Clone a sample Java e-commerce application from GitHub
- Write a multi-stage Dockerfile with separate **build** and **runtime** stages
- Understand dependency caching strategies with Maven
- Troubleshoot and fix a Docker build error related to JAR file naming
- Build and run the containerized Java application
- Verify the application is working via browser

### Step 2: Create the Multi-Stage Dockerfile

A multi-stage Dockerfile is created to separate the build and runtime environments:

```bash
nano Dockerfile
```

**Dockerfile contents:**

```dockerfile
# Multi-stage build for Spring Boot application

# Stage 1: Build stage
FROM maven:3.9.9-eclipse-temurin-21 AS builder

LABEL maintainer="sakitazeyek@gmail.com"

# Set the working directory inside the container
WORKDIR /app

# Copy the pom.xml to the container first (for better layer caching)
COPY pom.xml .

# Download dependencies (this layer will be cached if pom.xml doesn't change)
RUN mvn dependency:go-offline -B

# Copy the source code to the container
COPY src ./src

# Build the application using Maven
RUN mvn package -DskipTests

# Stage 2: Runtime stage
FROM eclipse-temurin:21-jre-alpine AS runtime

LABEL maintainer="Sakit"

# Create a non-root user for security
RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup

# Set the working directory inside the container
WORKDIR /app

# Copy the JAR file from the builder stage
COPY --from=builder /app/target/*.jar app.jar

# Change ownership of the app directory to the non-root user
RUN chown -R appuser:appgroup /app

# Switch to the non-root user
USER appuser

# Expose the port the app runs on
EXPOSE 8080

# Command to run the application
CMD ["java", "-jar", "app.jar"]
```

![Multi-Stage Dockerfile](Screenshots/Screenshot%202026-03-31%20082806.png)

**Stage 1 — Build Stage Breakdown:**

| Instruction | Purpose |
|---|---|
| `FROM maven:3.9.9-eclipse-temurin-21 AS builder` | Use Maven with JDK 21 as the build environment |
| `WORKDIR /app` | Set the working directory |
| `COPY pom.xml .` | Copy dependency file first for layer caching |
| `RUN mvn dependency:go-offline -B` | Download all dependencies (cached if pom.xml unchanged) |
| `COPY src ./src` | Copy source code |
| `RUN mvn package -DskipTests` | Build the JAR file, skipping tests |

**Stage 2 — Runtime Stage Breakdown:**

| Instruction | Purpose |
|---|---|
| `FROM eclipse-temurin:21-jre-alpine AS runtime` | Use lightweight JRE-only Alpine image |
| `RUN addgroup ... && adduser ...` | Create a non-root user for security |
| `COPY --from=builder /app/target/*.jar app.jar` | Copy only the built JAR from the build stage |
| `RUN chown -R appuser:appgroup /app` | Set proper ownership |
| `USER appuser` | Run as non-root user |
| `EXPOSE 8080` | Document the application port |
| `CMD ["java", "-jar", "app.jar"]` | Start the Spring Boot application |

### Step 3: Troubleshoot Build Error — JAR File Not Found

The initial build attempt failed due to a mismatch in the JAR filename:

![Build Error](Screenshots/Screenshot%202026-03-31%20083448.png)

**Error:**
```
ERROR [runtime 4/5] COPY --from=builder /app/target/ecommerce-app-1.1-SNAPSHOT.jar app.jar:
failed to build: failed to solve: failed to compute cache key: failed to calculate checksum
```

**Root Cause:** The original `COPY` instruction used a hardcoded filename (`ecommerce-app-1.1-SNAPSHOT.jar`), but the actual JAR filename generated by Maven may differ.

**Fix:** Use a wildcard pattern to copy the JAR file regardless of its exact name:

```dockerfile
# Before (hardcoded — caused error):
#COPY --from=builder /app/target/ecommerce-app-1.1-SNAPSHOT.jar app.jar

# After (wildcard — fixes the issue):
COPY --from=builder /app/target/*.jar app.jar
```

![Dockerfile Fix](Screenshots/Screenshot%202026-03-31%20083520.png)

> **Lesson Learned:** Using wildcard patterns (`*.jar`) makes the Dockerfile more resilient to version changes in the artifact name defined in `pom.xml`.

### Step 4: Build the Docker Image and Run the Container

After fixing the Dockerfile, rebuild the image and run the container:

```bash
# Build the Docker image
docker build -t mrquiet24/java-app-ms .

# Run the container, mapping port 80 to 8080
docker run -d -p 80:8080 mrquiet24/java-app-ms

# Verify the container is running
docker ps
```

![Successful Build and Run](Screenshots/Screenshot%202026-03-31%20084239.png)

**Key Observations:**
- Build completed successfully: `Building 1.4s (17/17) FINISHED`
- Many layers were **CACHED** from the failed build (dependency download, source copy, etc.), demonstrating Docker's build cache efficiency
- The container is running and mapped: `0.0.0.0:80->8080/tcp`
- Image is tagged as `mrquiet24/java-app-ms:latest`

**Port Mapping:**

| Host Port | Container Port | Purpose |
|---|---|---|
| `80` | `8080` | Access the app via `localhost` (port 80) on the host |

## 🏗️ Architecture — Multi-Stage Build

```
┌─────────────────────────────────────────────────────────┐
│                  MULTI-STAGE BUILD                      │
│                                                         │
│  ┌─────────────────────────────┐                        │
│  │   Stage 1: BUILD (builder)  │                        │
│  │   maven:3.9.9-temurin-21   │                        │
│  │                             │                        │
│  │   pom.xml → mvn install     │                        │
│  │   src/ → mvn package        │                        │
│  │   Output: target/*.jar      │───┐                    │
│  └─────────────────────────────┘   │                    │
│                                    │  COPY --from       │
│  ┌─────────────────────────────┐   │                    │
│  │  Stage 2: RUNTIME           │◄──┘                    │
│  │  eclipse-temurin:21-jre     │                        │
│  │                             │                        │
│  │  app.jar (only)             │                        │
│  │  Non-root user: appuser     │                        │
│  │  EXPOSE 8080                │                        │
│  └─────────────────────────────┘                        │
│                                                         │
│  Result: Small, secure runtime image                    │
│  (No Maven, no source code, no build tools)             │
└─────────────────────────────────────────────────────────┘
```

## 💡 Key Takeaways

1. **Multi-stage builds** dramatically reduce image size by separating the build environment (Maven + JDK) from the runtime (JRE only)
2. **Dependency caching** (`COPY pom.xml` before `COPY src`) ensures that dependencies are only re-downloaded when `pom.xml` changes, significantly speeding up rebuilds
3. **Non-root users** in Docker containers follow the **principle of least privilege**, improving security
4. **Wildcard patterns** in `COPY` instructions (`*.jar`) make Dockerfiles more maintainable and resilient to version changes
5. **Alpine-based images** (`eclipse-temurin:21-jre-alpine`) are significantly smaller than full OS images, reducing attack surface and pull times
6. **Build cache** allows Docker to skip unchanged layers, making iterative development and debugging faster
